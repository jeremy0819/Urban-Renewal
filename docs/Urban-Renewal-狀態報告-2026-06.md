## 一、執行摘要 Executive Summary

**一句話：** 本專案已從「3 份 markdown 方法論」成長為一套**可上線、零依賴、經事實查證**的都更／危老**數位決策與訓練體系**，涵蓋知識庫、互動儀表板、策略沙盤遊戲、財務試算與對外簡報；核心缺口在於**遊戲真實感調校**與**與計算工具（RE-DCF-Tool）的串接**。

- **完成度（交付物）：** 高。5 個自含式網頁（2,583 行）＋ 3 份方法論文件＋白皮書（Word）＋說明會簡報（PPT）＋ 5 份開發 spec，全部可離線開、可上 GitHub Pages。
- **品質與資安：** 良好。純前端、零外送、無密鑰、無 XSS、外連皆具 `noopener`、`.gitignore` 鐵律生效、洩漏掃描乾淨。
- **內容正確性：** 經查證。法規條號（§37／§65／§67／§162）、稅制（房地合一 2.0／囤房稅 2.0）、淨零（BERS）、2026 房市趨勢均已對照官方與媒體來源（見附錄 A）。
- **主要風險／缺口：** ① 策略沙盤「人最難整合」的張力尚未充分呈現（釘子戶不夠黏、難度偏易）；② 競爭建商系統目前為展示性（無實際敗局觸發）；③ 財務系統單向、與正式計算工具 RE-DCF-Tool 尚未串接。
- **建議下一步：** 先做低成本高回報的「真實感調校包」，再進入 RE-DCF-Tool **Phase 0/1 串接**（公式同源 → 參數交換），逐步邁向 V4–V7 路線圖。

## 二、專案現況與定位 Situation & Positioning

`Urban-Renewal` 為**方法論主庫（知識層）**，與姊妹庫 **RE-DCF-Tool（計算層，Streamlit）** 分工。本庫將「拿到一塊地 → 交屋」的都更／危老開發拆為 **S1–S11 全生命週期**，並以**不動產研析**為脊椎，向上收斂為一套可定位、可核對、可訓練的體系。

本期（單一工作週期）內，專案經歷 **18 次提交**，由純文件演進為**四層交付**：
1. **知識層**：開發流程架構、投報分析正典、整合開發評估總論。
2. **互動層**：開發儀表板（雙模式）、整合策略沙盤（v4）、坪效・開發試算。
3. **對外層**：住戶說明會簡報（HTML＋PPT）。
4. **工程層**：5 份 spec、2 支文件產生器、資安與洩漏掃描。

## 六、關鍵發現 Key Findings（含實測缺口）

### 強項 Strengths
1. **內容經查證、法規標條號**，專業模式可供從業者直接引用。
2. **零依賴、可離線、可上線**，維運與資安成本極低。
3. **雙模式分層**同時服務外行（白話）與專業（條號／公式）。
4. **沙盤的人格／家族／情緒機制已落地且可驗證**，把「都更最難的是人」具象化。

### 缺口 Gaps（依影響排序，皆為實測所得）
| # | 發現 | 證據 | 影響 |
|---|---|---|---|
| G1 | **釘子戶不夠黏、難度偏易** | 實測第 3 回合即清空反對／敵對，第 8 回合達 **100% 支持** | 核心賣點「人最難整合」感受不足 |
| G2 | **競爭建商為展示性** | 擺爛測試三場皆「時間到」而敗，**無一場由對手達標觸發** | 競爭機制目前無實質張力 |
| G3 | **財務單向、多結局形同虛設** | ROI 僅隨時間下滑（32%→23%），無正向槓桿；每局皆落「完美整合」 | 財務面缺乏可優化變數 |
| G4 | **回饋斷層** | 觀望派 bandwagon 隱形、達標後未催收尾、單卡（說明會）近萬用 | UX 與教學效果受限 |

## 七、建議與執行展望 Recommendations & Roadmap

### 7.1 即刻優先（低成本、高回報）— 「真實感調校包」
直接修 G1／G2／G4，最對準專案靈魂：
- **釘子戶黏性**：植入少數高抗性樞紐戶，僅特定高成本動作可撬動；門檻達標後仍保留殘餘 holdout（靠權變強制）。
- **競爭建商長牙**：對手速度與「你的怠惰」連動，使「被搶先」成為真實敗局。
- **回饋補強**：bandwagon 即時提示、達標催收尾、情境人格組成逼出不同最優解。

### 7.2 中期 — RE-DCF-Tool 串接（修 G3）
RE-DCF-Tool 經查證為 **Streamlit Python**（L1–L6：§162 逐層容積查核、坪效、都更全案六科目投報、敏感度 heatmap、三組黃金測試），**無對外 API**。據此分三階段：

| 階段 | 做法 | 改 RE-DCF？ | 價值 |
|---|---|---|---|
| **Phase 0** 公式同源 | 沙盤財務面板＋evaluator 採與 RE-DCF 相同的 L2–L6 公式骨架；雙向 cross-link | 零改 | 立即一致；ROI 取得正向槓桿（容積獎勵申請、模式選擇） |
| **Phase 1** 參數交換 | 定義「都更案件參數交換 JSON」；前端深連結（`st.query_params`）或 JSON 匯入匯出 | 小改 | 沙盤／試算 → RE-DCF 一鍵精算 |
| **Phase 2** 計算 API | RE-DCF 旁加 FastAPI／/api 回 KPI／IRR JSON；前端取回 | 較大（需後端＋CORS） | 啟用案件生成器、自動化（接 V5／V6） |

> **資安鐵則：** RE-DCF 之真實案夾已 gitignore；任何串接一律「使用者輸入、不落地公庫」。建議追加**「真實數字本地模式」**（僅存瀏覽器、永不上傳）供內訓安全使用真實參數。

### 7.3 長期 — V4→V7 路線圖（沿用既有 md）
- **V4（進行中）**：財務系統 ✅、競爭建商（待長牙）、法規風險擴充 ✅。
- **V5**：RE-DCF 串接、案件生成器（使用者輸入生成通用劇本）。
- **V6**：AI 地主代理人（需 LLM 後端，靜態頁無法承載，須改架構）。
- **V7**：都市更新數位決策平台（知識庫＋計算工具＋AI 地主整合）。

### 7.4 上線與治理 Outlook
- **上線**：請於 GitHub **Settings → Pages → main /（root）** 啟用，即得 `jeremy0819.github.io/Urban-Renewal/`。
- **建議 KPI**：開發部試用回收（沙盤難度合理性、專業 tag 精準度）、Pages 流量、訓練覆盤產出數。
- **風險**：①AI 地主需後端投入；②真實數字一律本地、嚴守不入庫；③Google Fonts 第三方請求可選擇自託管消除。
